Veille technologique

Mon sujet de veille, mes sources, et son application concrète dans mes projets.

Sujet de veille

J'ai choisi de faire ma veille sur la sécurité du développement web, et plus précisément sur les attaques classiques contre les sites en PHP/MySQL : injections SQL, XSS, gestion des sessions, mots de passe.

Pourquoi ce sujet :

  • C'est un fondamental du métier : on l'aborde rapidement en cours dès la première année
  • C'est un sujet qui restera utile longtemps, ce n'est pas une mode passagère
  • En sécurité, il faut continuer à se mettre à jour : de nouvelles vulnérabilités apparaissent régulièrement

J'ai choisi ce sujet dès la première année du BTS, et il s'est avéré particulièrement utile lors de mon stage de deuxième année sur Websciences, qui gère des comptes utilisateurs et des contenus payants.

Mes sources

J'utilise plusieurs types de sources pour croiser les points de vue :

Sources institutionnelles

  • OWASP (Open Worldwide Application Security Project) : la référence mondiale sur la sécurité web, avec son célèbre Top 10 des failles les plus critiques
  • ANSSI (Agence nationale de la sécurité des systèmes d'information) : les guides officiels français, dont les bonnes pratiques pour le développement web
  • CERT-FR : les alertes de sécurité publiées par l'État

Documentation officielle

  • php.net : la doc de PHP, surtout les pages sur PDO, password_hash et les sessions
  • MDN Web Docs (Mozilla) pour le HTML, CSS et JavaScript

Médias spécialisés

  • Le Journal du Hacker et Korben.info pour suivre l'actualité sécurité en français
  • Stack Overflow et les blogs techniques (Cloudflare, GitHub) pour des retours d'expérience concrets

Mon outillage

Je centralise tout dans un agrégateur RSS (Inoreader), qui me regroupe les nouveaux articles. J'ai aussi des marque-pages classés par thème, et je note dans un carnet ce que je veux retenir et les questions que je me pose.

Ce que j'en retiens

Quelques règles qui reviennent toujours

Quel que soit le sujet précis, certains principes reviennent systématiquement :

  1. Ne jamais faire confiance aux données qui viennent du navigateur. Toute donnée envoyée par un utilisateur doit être vérifiée avant d'être utilisée.
  2. Privilégier les protections automatiques. Par exemple, une requête préparée protège mieux qu'un échappement à la main, parce qu'on ne peut pas l'oublier.
  3. Limiter ce qu'on expose. Moins une fonctionnalité a de paramètres, moins il y a de risques qu'elle soit détournée.
  4. Garder une trace. Sans logs, on ne voit pas une attaque arriver et on ne peut pas comprendre ce qui s'est passé.

Trois sujets que j'ai creusés

Les injections SQL. Encore aujourd'hui, c'est une des principales causes de fuites de données. Pourtant, la solution est simple : utiliser des requêtes préparées. Ce que j'en retiens, c'est que ce n'est pas un problème technique mais une question d'habitude : il faut prendre le réflexe de ne jamais mettre une variable directement dans une requête SQL.

Le stockage des mots de passe. J'ai lu plusieurs rapports sur des fuites de données et j'ai été surpris du nombre de bases où les mots de passe sont stockés en clair ou avec des algos dépassés (MD5, SHA1). En PHP, password_hash() fait le boulot tout seul. Sur Websciences, je l'utilise systématiquement.

Les sessions. J'ai pris du temps pour comprendre comment fonctionne $_SESSION en PHP, le rôle des cookies, et les protections contre le vol de session (régénérer l'identifiant à la connexion, mettre l'attribut HttpOnly sur les cookies, prévoir une expiration). C'est un domaine où le code peut sembler fonctionner sans avoir les bonnes protections.

Application concrète sur Websciences

Quand est venu le moment de développer Websciences, cette veille m'a permis de mettre en place les bonnes pratiques sans hésiter :

Requêtes préparées partout

Avec PDO, je n'ai jamais collé directement une variable dans une requête SQL. À la place, chaque valeur passe par un paramètre. Ce choix a été pris dès le début, pas en correctif.

password_hash et password_verify

Les mots de passe ne sont jamais stockés en clair. PHP s'occupe du hachage automatiquement, avec l'algorithme bcrypt qui est solide.

Contrôle d'accès uniquement côté serveur

Pour les ressources réservées aux abonnés, c'est PHP qui décide si on peut y accéder ou non, jamais le JavaScript. Comme ça, un utilisateur ne peut pas contourner la limite en bidouillant le code de la page dans son navigateur.

Pour la suite

Ce que cette veille m'a appris, c'est que la sécurité n'est pas qu'une question technique : derrière les bases de données, il y a de vraies personnes dont les informations peuvent être volées et utilisées contre elles. Je vois ça maintenant comme une responsabilité du développeur, pas comme une contrainte.

Pour la suite, j'aimerais élargir ma veille à deux sujets : les en-têtes de sécurité HTTP (Content-Security-Policy par exemple) et la conteneurisation avec Docker.
