Plateforme Websciences

Concevoir et développer une plateforme web complète de ressources pédagogiques en architecture MVC PHP/MySQL — du modèle de données au déploiement local.

Contexte et besoins

Pendant ma première année, j'avais déjà mis en place un serveur Discord pour Mention TB. Mais Discord, c'est bien pour discuter et s'entraider, ce n'est pas fait pour ranger et retrouver des ressources pédagogiques. Les fiches de cours et d'exercices continuaient à circuler comme des pièces jointes, sans aucun classement.

L'agence voulait donc un site web dédié, sur lequel les enseignants pourraient mettre en ligne leurs ressources, organisées par niveau (Collège, Seconde, Première, Terminale) et par matière (Maths, Physique, Chimie). L'idée était aussi de proposer un abonnement à 7 € par mois pour accéder aux corrigés et télécharger les fiches en PDF.

Voici le cahier des charges qu'on a validé ensemble avec mon tuteur :

Construire un site web qui met à disposition cours, exercices et corrigés en maths et physique-chimie pour les niveaux Seconde, Première et Terminale, avec un accès gratuit aux cours et exercices et un accès payant aux corrigés et aux téléchargements.

Choix techniques

Avant de me lancer, j'ai discuté des choix techniques avec mon tuteur. Vu le contexte (petite agence, stagiaire, deux mois), il fallait rester sur des technos simples et maintenables.

PHP 8 MySQL 8 Architecture MVC HTML 5 CSS 3 JavaScript Apache (Laragon) .htaccess Python 3 ReportLab Git VS Code

PHP plutôt qu'un framework

En BTS SIO, on apprend PHP en programmation orientée objet, mais pas de framework comme Symfony ou Laravel. J'ai donc choisi de construire une architecture MVC en PHP « pur », sans framework. C'est plus formateur (on comprend mieux comment ça marche), et ça reste à la portée de quelqu'un qui aurait à reprendre le projet plus tard.

MySQL pour la base de données

MySQL est la base de données qu'on a vue en BTS. Côté code, j'utilise PDO, l'outil PHP standard pour parler à MySQL, en passant uniquement par des requêtes préparées. C'est ce qui permet d'éviter les injections SQL (j'en parle plus dans la section sécurité plus bas).

Laragon comme environnement local

Je développe et fais tourner le site sur Laragon, qui installe Apache, PHP et MySQL d'un coup sur Windows. C'est plus simple qu'installer chaque composant séparément. La mise en ligne sur un vrai hébergeur n'était pas au programme du stage, ce sera une étape pour plus tard.

Architecture MVC

L'architecture suit une séparation stricte des trois couches Modèle / Vue / Contrôleur, complétée par un répertoire core/ pour les classes utilitaires (connexion à la base de données, routeur, contrôleur abstrait, gestion des sessions) et un répertoire config/ pour la configuration et les scripts SQL.

Le routage des URL

Toutes les requêtes du site passent par public/index.php grâce à une règle dans le fichier .htaccess qui réécrit les URL. Par exemple, quand un élève arrive sur /cours/seconde/fonctions, le code regarde l'URL et appelle le bon contrôleur avec les bons paramètres. L'avantage, c'est d'avoir des URL lisibles (mieux pour les utilisateurs et pour Google), au lieu d'URL du genre ?page=cours&id=42.

Le pattern Repository

Toutes les requêtes SQL sont regroupées dans des classes Repository, plutôt que dispersées un peu partout dans le code. Pour chaque table (User, Resource, Chapter), il y a une classe entité qui représente les données et une classe repository qui contient les méthodes pour les chercher, les ajouter ou les modifier en base.

Quelques extraits de code

Voici deux extraits qui illustrent le fonctionnement de la plateforme.

Extrait CSS — Variables de la charte graphique

Toutes les couleurs et polices du site sont définies une seule fois dans des variables CSS, en haut de la feuille de style. C'est ce qui permet d'avoir une charte cohérente sur toutes les pages : si on veut changer la couleur dorée, on la modifie une seule fois et ça se répercute partout.

/* style.css — palette et typographie de la plateforme */
:root {
    /* Couleurs */
    --bg-deep:    #0a0a0c;     /* fond principal */
    --bg-card:    #1a1a20;     /* fond des cartes  */
    --gold:       #d4a84b;     /* doré principal  */
    --gold-soft:  rgba(212, 168, 75, 0.12);
    --text:       #e8e6df;     /* texte principal */
    --text-soft:  #b8b6ae;     /* texte secondaire */

    /* Polices */
    --font-serif: 'Cormorant Garamond', serif;
    --font-sans:  'Outfit', sans-serif;
}

/* Utilisation : on appelle les variables avec var() */
.bouton-or {
    background: var(--gold);
    color: var(--bg-deep);
    font-family: var(--font-sans);
}

Extrait PHP — Vérification du mot de passe à la connexion

Quand un utilisateur se connecte, son mot de passe ne doit jamais être comparé en clair avec ce qui est en base. PHP fournit deux fonctions standards : password_hash() pour stocker le mot de passe (lors de l'inscription) et password_verify() pour vérifier qu'un mot de passe saisi correspond bien à ce qui est stocké.

// AuthController.php — extrait de la fonction de connexion
public function login($email, $motDePasse) {
    // On va chercher l'utilisateur par son email
    $sql = "SELECT * FROM users WHERE email = :email";
    $stmt = $this->db->prepare($sql);
    $stmt->execute([':email' => $email]);
    $user = $stmt->fetch();

    // On vérifie que le mot de passe correspond
    if ($user && password_verify($motDePasse, $user['password'])) {
        $_SESSION['user_id'] = $user['id'];
        return true;
    }
    return false;
}

Sécurité applicative

J'ai pris en compte la sécurité dès le début du projet, parce que c'est aussi le sujet de ma veille technologique. Trois choses à retenir :

  1. Requêtes préparées partout. Comme dans l'extrait ci-dessus, on ne met jamais une variable directement dans une requête SQL. À la place, on utilise un paramètre (:email) qui est ensuite remplacé proprement. C'est ce qui empêche les injections SQL, une des attaques les plus courantes sur le web.
  2. Mots de passe hachés. Les mots de passe ne sont jamais stockés en clair. password_hash() les transforme en une suite de caractères impossible à inverser, et password_verify() permet de comparer sans avoir à connaître le mot de passe d'origine.
  3. Contrôle d'accès côté serveur. Pour les corrigés (réservés aux abonnés), la vérification se fait dans le contrôleur PHP, pas en JavaScript dans le navigateur. Ça évite qu'un utilisateur puisse contourner la limite en bidouillant le code de la page.

Génération automatique des fiches PDF

Particularité du projet : les fiches pédagogiques (cours, exercices, corrigés) sont générées automatiquement par un script Python qui utilise la bibliothèque ReportLab. Ça permet d'avoir des PDF à la mise en page identique d'une fiche à l'autre : même police, mêmes couleurs, même en-tête, même pied de page.

Les PDF générés sont enregistrés dans public/pdfs/, et la base de données ne stocke que le chemin du fichier (dans la colonne pdf_path de la table resources). Comme ça, on sépare bien les métadonnées (titre, niveau, matière) du fichier lui-même, ce qui est plus propre.

ReportLab n'est pas au programme du BTS, j'ai dû me former dessus tout seul. C'était plutôt formateur même si la documentation n'est pas toujours évidente, surtout pour afficher des symboles mathématiques (ℕ, ℤ, ℚ, ℝ) ou des fractions verticales.

Recette et tests

Avant de livrer, j'ai testé tous les parcours utilisateurs principaux :

  • Inscription d'un nouvel utilisateur (avec validation de l'email et d'un mot de passe assez robuste)
  • Connexion et déconnexion, et le fait que la session reste active entre deux pages
  • Navigation par niveau, par matière, par chapitre
  • Accès libre à un cours, accès libre à un exercice
  • Tentative d'accès à un corrigé sans abonnement → redirection vers la page d'abonnement
  • Accès à un corrigé avec un compte abonné
  • Téléchargement d'un PDF

J'ai testé chaque cas à la main et noté le résultat dans un tableau (résultat attendu / résultat observé). Quand il y avait un écart, je corrigeais le code avant de continuer.

Difficultés rencontrées

Le routage des URL sans framework

La grosse difficulté technique a été de faire fonctionner les URL propres. Sans framework, il faut comprendre toute la chaîne : la requête arrive sur Apache, qui regarde le fichier .htaccess, qui redirige vers index.php, qui ensuite analyse l'URL pour appeler le bon contrôleur. Au début, ça ne marchait pas parce que j'avais oublié d'activer le module mod_rewrite dans Apache (il est désactivé par défaut). La documentation Apache et plusieurs heures de tests m'ont permis de comprendre.

L'apprentissage de ReportLab

ReportLab est puissant mais sa documentation n'est pas toujours claire pour les cas un peu avancés. Pour afficher des symboles mathématiques ou des fractions, j'ai dû tâtonner et tester beaucoup de choses avant de trouver des solutions qui marchent.

Garder une charte cohérente entre site et PDF

Pour que le site et les PDF aient le même look, j'ai défini une charte commune (mêmes couleurs noir/or, mêmes polices, mêmes en-têtes) et je l'ai appliquée des deux côtés.

Bilan

Au bout des huit semaines, la plateforme Websciences fonctionne en local sur le périmètre prévu pour le stage. Les chapitres du niveau Seconde sont les plus avancés et servent de jeu de référence ; ceux de Première et Terminale continuent à être rédigés par les profs après mon stage.

Ce stage m'a beaucoup apporté :

  • Une vraie compréhension du MVC, parce que je l'ai construit moi-même au lieu d'utiliser un framework qui le cache
  • De l'autonomie sur des sujets pas vus en cours (ReportLab, configuration d'Apache)
  • Une vraie attention à la sécurité, qui n'est pas juste un détail mais quelque chose à intégrer dès le début
