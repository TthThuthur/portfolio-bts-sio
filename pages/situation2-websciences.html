Plateforme Websciences

Concevoir et développer une plateforme web complète de ressources pédagogiques en architecture MVC PHP/MySQL — du modèle de données au déploiement local.

Contexte et expression des besoins

Au cours de la première année, j'avais doté l'agence Mention TB d'un espace de communication via Discord. Au terme de cette première mission, un constat est apparu : Discord est efficace pour l'entraide et la prise de rendez-vous, mais il n'est pas adapté à l'archivage et à la consultation structurée de ressources pédagogiques. Les fiches de cours et d'exercices continuaient à circuler comme des pièces jointes éphémères, sans logique de classement.

L'agence souhaitait une plateforme web dédiée, qui hébergerait l'ensemble des ressources produites par les enseignants, accessibles aux élèves selon une logique de niveau (Collège, Seconde, Première, Terminale) et de matière (Mathématiques, Physique, Chimie). La plateforme devait également permettre une monétisation légère via un abonnement de 7 € par mois, donnant accès aux corrigés et au téléchargement PDF.

Voici la formulation initiale du cahier des charges, validée avec mon tuteur de stage :

Construire une plateforme web qui mette à disposition cours, exercices et corrigés en mathématiques et physique-chimie pour les niveaux Seconde, Première et Terminale, avec un accès gratuit aux cours et exercices et un accès payant aux corrigés et téléchargements.

Choix technologiques

Avant de me lancer, j'ai pris le temps d'arbitrer les choix techniques avec mon tuteur. Le contexte (petite agence, stagiaire, durée limitée) imposait des choix sobres et maintenables.

PHP 8 MySQL 8 Architecture MVC PDO HTML 5 CSS 3 JavaScript Apache (Laragon) .htaccess Python 3 ReportLab Git VS Code

PHP natif plutôt qu'un framework

Le BTS SIO m'a formé à PHP en programmation orientée objet, mais pas à un framework comme Symfony ou Laravel. J'ai donc choisi de bâtir une architecture MVC depuis zéro, sans framework. Ce choix présente trois avantages dans le cadre d'un stage : il me force à comprendre comment fonctionne réellement un MVC (routage, autoloading, séparation des couches), il maintient le projet à la portée d'un mainteneur futur formé au même niveau, et il évite de masquer derrière des conventions automatiques des concepts que le BTS demande de maîtriser explicitement.

MySQL et PDO

MySQL est la base de données enseignée pendant le BTS. J'ai utilisé l'extension PDO avec exclusivement des requêtes préparées, pour deux raisons : protection contre les injections SQL, et possibilité de changer de SGBD ultérieurement sans réécrire la couche d'accès aux données.

Laragon comme environnement local

Le projet est développé et exécuté sur Laragon, qui regroupe Apache, PHP et MySQL dans un environnement Windows simple à configurer. Le déploiement vers un hébergement distant n'a pas été abordé dans la formation à ce stade : il est prévu comme étape suivante mais ne fait pas partie du périmètre de ce stage.

Le routeur

Toutes les requêtes entrantes passent par public/index.php grâce à une règle .htaccess qui réécrit les URL. Le routeur analyse le chemin demandé, extrait éventuellement des paramètres (par exemple /cours/seconde/fonctions), et instancie le contrôleur approprié.

// core/Router.php — extrait simplifié
public function dispatch(string $uri): void
{
    $uri = trim(parse_url($uri, PHP_URL_PATH), '/');
    foreach ($this->routes as $pattern => $handler) {
        if (preg_match($pattern, $uri, $matches)) {
            [$controller, $method] = explode('@', $handler);
            (new $controller())->$method(...array_slice($matches, 1));
            return;
        }
    }
    throw new NotFoundException();
}

Le pattern Repository

Toutes les requêtes SQL sont centralisées dans des classes Repository, jamais dispersées dans les contrôleurs ou les vues. Chaque entité métier (User, Resource, Chapter) a une entité simple (les attributs, le constructeur) et un repository (les méthodes d'accès aux données).

// model/ResourceRepository.php — extrait
public function findByChapter(int $chapterId): array
{
    $sql = "SELECT * FROM resources
            WHERE chapter_id = :id
            ORDER BY type";
    $stmt = $this->db->prepare($sql);
    $stmt->execute([':id' => $chapterId]);

    return array_map(
        fn($row) => new Resource($row),
        $stmt->fetchAll()
    );
}

Sécurité applicative

La sécurité a été intégrée dès le départ — pas comme une couche ajoutée à la fin. Trois mesures principales :

  1. Requêtes préparées partout. Aucune concaténation directe de variables dans une requête SQL. Toutes les valeurs passent par des paramètres nommés via PDO. Cela élimine la classe d'attaques par injection SQL.
  2. Hachage des mots de passe avec password_hash() et l'algorithme par défaut de PHP (bcrypt). La vérification utilise password_verify(). À aucun moment un mot de passe n'est stocké ou comparé en clair.
  3. Contrôle d'accès dans le contrôleur. Les ressources « subscriber » ne sont servies au navigateur que si Auth::user()->hasRole('subscriber') retourne vrai. La vérification ne dépend ni du JavaScript ni d'un paramètre de requête manipulable côté client.

Génération automatique des fiches PDF

Une particularité du projet est que les fiches pédagogiques (cours, exercices, corrigés) sont générées par programme à partir d'un script Python utilisant la bibliothèque ReportLab. Cette chaîne, développée en parallèle de la plateforme, garantit une charte graphique uniforme entre toutes les fiches : police, couleurs, mise en page, en-têtes, pieds de page.

Les PDF générés sont déposés dans public/pdfs/, et la base de données stocke uniquement leur chemin via la colonne pdf_path de la table resources. Cette séparation entre métadonnées (en base) et fichiers (sur disque) est une bonne pratique pour la performance et la sauvegarde.

L'apprentissage de ReportLab a été l'un des aspects les plus formateurs du stage : c'est une bibliothèque puissante mais peu documentée pour les usages avancés (rendu de symboles mathématiques, tableaux de variations, fractions verticales), et j'ai dû développer mes propres petits composants réutilisables.

Recette et tests

Avant la livraison, j'ai mis en place un jeu d'essai conséquent couvrant les principaux parcours :

  • Inscription d'un nouvel utilisateur, validation du formulaire (email, mot de passe robuste)
  • Connexion et déconnexion, persistance de la session
  • Navigation par niveau, par matière, par chapitre
  • Accès libre à un cours, accès libre à un exercice
  • Tentative d'accès à un corrigé sans abonnement → redirection vers la page d'abonnement
  • Accès à un corrigé après simulation d'un compte abonné
  • Téléchargement d'un PDF

Chaque cas a été exécuté manuellement et documenté dans une grille de recette, avec le résultat attendu et le résultat observé. Les écarts ont fait l'objet d'un correctif avant la livraison.

Difficultés rencontrées

Le routage sans framework

La principale difficulté technique a été la mise en place du routage propre. Sans framework, il faut comprendre la chaîne complète : la requête arrive sur Apache, qui consulte le .htaccess, qui réécrit l'URL vers index.php, qui à son tour analyse l'URL d'origine pour appeler le bon contrôleur. La première version ne fonctionnait pas car j'avais oublié d'activer mod_rewrite dans Apache, qui est désactivé par défaut. La documentation officielle d'Apache et plusieurs heures de tests ont permis de débloquer le problème.

L'apprentissage de ReportLab

ReportLab est très puissant mais peu documenté pour les cas avancés. La génération de symboles mathématiques (ℕ, ℤ, ℚ, ℝ), de fractions verticales et de tableaux de variations a nécessité de créer mes propres « flowables » personnalisés. J'ai progressé par itérations, en m'inspirant de fragments trouvés dans la documentation officielle et en testant à chaque étape.

La cohérence d'identité visuelle

Maintenir une identité visuelle cohérente entre la plateforme web (HTML/CSS) et les PDF générés (ReportLab) a demandé de la rigueur. J'ai défini une charte commune : palette noir profond / or / blanc cassé, polices serif pour les titres et sans-serif pour le corps, mise en page sobre. Cette cohérence visuelle est ce qui donne à la plateforme son caractère professionnel.

Bilan

À l'issue des huit semaines, la plateforme Websciences est fonctionnelle en environnement local. Elle couvre les niveaux Seconde, Première et Terminale (les chapitres de Seconde sont les plus complets, c'est le périmètre cible du stage), avec un accès différencié gratuit/abonné et une chaîne de génération PDF utilisable.

Au-delà du livrable, ce stage m'a apporté trois acquis majeurs :

  • Une compréhension profonde de l'architecture MVC, parce que j'ai dû la construire moi-même et non l'utiliser via un framework
  • Une véritable autonomie sur des sujets non couverts par la formation (ReportLab, configuration Apache)
  • Une conscience de la sécurité applicative, qui n'est plus pour moi un ajout cosmétique mais une dimension structurante du développement
